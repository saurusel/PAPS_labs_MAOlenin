## 1. Проектные решения

Ниже перечислены основные решения, которые были приняты при проектировании учебного API “внутренний мерч‑магазин” (каталог товаров + заказы). Формулировки сделаны так, чтобы было понятно **что выбрано**, **как это работает** и **зачем именно так**.

1. **REST‑подход и ресурсо‑ориентированные пути.**  
   API построен вокруг ресурсов (существительных), а не “действий”: `/products`, `/orders`. Это делает интерфейс предсказуемым: клиент “работает с сущностями”, а операции выражаются HTTP‑методами.

2. **Семантика методов HTTP (GET/POST/PUT/DELETE).**  
   В API используются базовые методы:
   - `GET` — получить данные (списки/детали);
   - `POST` — создать ресурс (товар, заказ);
   - `PUT` — изменить состояние ресурса (статус заказа);
   - `DELETE` — отменить заказ (в учебной модели — перевод в статус `Отменен` с бизнес‑действиями: возврат поинтов и снятие резерва).

3. **Версионирование в URL через `/api/v1`.**  
   Версия включена в путь, чтобы при изменениях контракта можно было выпустить `/api/v2` и не ломать существующих клиентов. Для Postman и Swagger это также проще всего.

4. **Stateless‑взаимодействие.**  
   Сервер не хранит “сессии”; каждый запрос несёт всю необходимую информацию (роль/пользователь). Это соответствует принципам REST и облегчает тестирование.

5. **Упрощённая модель доступа через заголовок `X-Role`.**  
   В лабораторной реализовано разграничение прав без JWT/OAuth2: роль передаётся заголовком, а сервер проверяет право на операцию. Это позволяет явно продемонстрировать RBAC‑логику и легко тестировать её в Postman.

6. **Пользовательский контекст через `X-User-Id` и правило “buyer видит только своё”.**  
   Для роли `buyer` используется заголовок `X-User-Id`. Покупатель может создавать заказ и видеть/отменять **только свои** заказы. Это демонстрирует базовый принцип защиты данных пользователей даже в учебном сервисе.

7. **Единый формат ошибок (контракт ошибок — часть API).**  
   Все ошибки приводятся к единому JSON‑формату:
   ```json
   { "error": { "code": "SOME_CODE", "message": "…", "details": {} } }
   ```
   Благодаря этому клиенту проще обрабатывать ошибки, а Postman‑тесты могут стабильно проверять `error.code`.

8. **Валидация данных на границе системы (Pydantic) + `422`.**  
   Тело запросов проверяется схемами (Pydantic) и ограничениями (`min_length`, `ge`, `le`, `min_items`). Некорректный ввод возвращает `422` с `error.code = VALIDATION_ERROR` (включая детали ошибок).

9. **In‑memory хранилище вместо БД (допущение лабораторной).**  
   Для упрощения запуска и фокуса на проектировании API данные хранятся в памяти процесса (`dict/list`). Это нормально для ЛР: главное — контракт API, статус‑коды, бизнес‑правила и тестирование.

10. **SKU как ключ варианта товара и запрет дубликатов.**  
   Заказ ссылается на варианты по `sku`, поэтому SKU обязаны быть уникальными. При создании товара сервер:
   - не допускает повторяющиеся SKU внутри одного запроса;
   - не допускает SKU, которые уже существуют в системе.

11. **Статусы заказа как конечный автомат (FSM) и запрет недопустимых переходов.**  
   Заказ живёт в ограниченном наборе статусов: `Новый`, `На проверке`, `Сборка`, `Готов к выдаче`, `Завершен`, `Отменен`.  
   Переходы контролируются на сервере (например, нельзя перескочить через этап), что защищает бизнес‑процесс от некорректных действий клиента.

12. **Согласованность поинтов и остатков (бизнес‑правила).**  
   - При создании заказа поинты списываются сразу (с записью в ledger).  
   - При переводе в `На проверке` выполняется резерв остатков.  
   - При отмене снимается резерв и возвращаются поинты.  
   - При переводе в `Завершен` списываются остатки (и резерв уменьшается).  
   Это создаёт понятную и проверяемую модель поведения заказа.

## 2. Документация API

## 2.0. Общая информация

- **Swagger UI:** `http://127.0.0.1:8000/docs`  
- **Base URL:** `http://127.0.0.1:8000/api/v1`  
- **Формат данных:** JSON (`Content-Type: application/json`)  
- **Заголовки (упрощённо для лабораторной):**
  - `X-Role: content_admin | buyer | fulfillment_admin`
  - `X-User-Id: <строка>` — требуется для части операций роли `buyer`

### Формат ошибок (единый)
```json
{
  "error": {
    "code": "SOME_CODE",
    "message": "Читаемое описание",
    "details": {}
  }
}
```

### Тестовые данные (для воспроизведения сценариев)
- Тестовые пользователи и балансы поинтов (in‑memory): `u1 = 5000`, `u2 = 2000`.
- Идентификация пользователя выполняется заголовком `X-User-Id`.
- Роли задаются заголовком `X-Role`.

### Матрица доступа (роль → endpoint)
- `content_admin`: `POST /products`
- `buyer`: `POST /orders`, `GET /orders` (только свои), `DELETE /orders/{id}` (только свои и только в ранних статусах)
- `fulfillment_admin`: `GET /orders` (все), `PUT /orders/{id}/status`, `DELETE /orders/{id}` (в ранних статусах)

### Часто используемые статус‑коды
- `200 OK` — успешное получение/обновление
- `201 Created` — ресурс создан
- `400 Bad Request` — бизнес‑ошибка (например, запрещённый переход статуса)
- `403 Forbidden` — недостаточно прав
- `404 Not Found` — ресурс/sku не найден
- `409 Conflict` — конфликт данных/состояния (SKU уже существует, недостаточно поинтов/остатков)
- `422 Unprocessable Entity` — ошибка валидации входных данных

## 2.1. Ресурс: Товары (products)

### 2.1.1. Создание товара
**Метод:** `POST`  
**URL:** `/products`  
**Заголовки:** `X-Role: content_admin`

**Что делает:** создаёт товар и список его вариантов (variants). Каждый вариант имеет `sku`, цену в поинтах и остаток.

**Формат запроса (Body):**
```json
{
  "name": "Hoodie",
  "description": "Тёплое худи с логотипом",
  "images": ["https://example.com/hoodie.png"],
  "variants": [
    { "sku": "HD-BLK-M", "size": "M", "color": "Black", "price_points": 1200, "stock_total": 10 },
    { "sku": "HD-WHT-L", "size": "L", "color": "White", "price_points": 1200, "stock_total": 5 }
  ]
}
```

**Что возвращает:** `201 Created` и созданный продукт.  
Варианты в ответе содержат также служебные поля `reserved` и `product_id`.

**Пример ответа:**
```json
{
  "id": 1,
  "name": "Hoodie",
  "description": "Тёплое худи с логотипом",
  "images": ["https://example.com/hoodie.png"],
  "variants": [
    {
      "sku": "HD-BLK-M",
      "size": "M",
      "color": "Black",
      "price_points": 1200,
      "stock_total": 10,
      "reserved": 0,
      "product_id": 1
    }
  ]
}
```

**Типовые ошибки:**
- `403 FORBIDDEN` — роль не `content_admin`
- `422 VALIDATION_ERROR` — повтор SKU внутри одного запроса
- `409 SKU_EXISTS` — SKU уже существует в системе

### 2.1.2. Получение списка товаров + поиск
**Метод:** `GET`  
**URL:** `/products`  
**Query (опционально):** `q` — подстрока для поиска по `name` (без учёта регистра)

**Что делает:** возвращает список товаров. Если передан `q`, список фильтруется по названию.

**Примеры запросов:**
- `/products`
- `/products?q=hood`

**Ответ:** `200 OK` и массив продуктов.

### 2.1.3. Получение товара по ID
**Метод:** `GET`  
**URL:** `/products/{product_id}`  
**Параметр пути:** `product_id` (int)

**Что делает:** возвращает один товар по идентификатору.

**Ответ:** `200 OK` и объект продукта.

**Типовые ошибки:**
- `404 NOT_FOUND` — товар не найден (`details.product_id` содержит id)

## 2.2. Ресурс: Заказы (orders)

### 2.2.1. Создание заказа
**Метод:** `POST`  
**URL:** `/orders`  
**Заголовки:** `X-Role: buyer`, `X-User-Id: <user>` (обязательно)

**Что делает (логика):**
1) Проверяет, что все `sku` из позиций существуют.  
2) Считает стоимость: `line_points = price_points * qty`, затем суммирует в `total_points`.  
3) Проверяет баланс поинтов пользователя.  
4) Списывает поинты сразу и создаёт заказ со статусом `Новый`.

**Формат запроса (Body):**
```json
{
  "items": [
    { "sku": "HD-BLK-M", "qty": 2 }
  ]
}
```

**Ответ:** `201 Created` и объект заказа.

**Структура заказа в ответе (ключевые поля):**
- `id`, `user_id`, `status`
- `items[]` с расчётом `unit_points`, `line_points`
- `total_points`
- `created_at`, `updated_at` (ISO‑8601 UTC со суффиксом `Z`)
- `history[]` (аудит переходов статусов)
- `reserved` (признак, выполнено ли резервирование остатков)

**Пример ответа:**
```json
{
  "id": 1,
  "user_id": "u1",
  "status": "Новый",
  "items": [
    { "sku": "HD-BLK-M", "qty": 2, "unit_points": 1200, "line_points": 2400 }
  ],
  "total_points": 2400,
  "created_at": "2026-02-26T10:00:00Z",
  "updated_at": "2026-02-26T10:00:00Z",
  "history": [
    { "ts": "2026-02-26T10:00:00Z", "from": null, "to": "Новый", "by_role": "buyer" }
  ],
  "reserved": false
}
```

**Типовые ошибки:**
- `403 FORBIDDEN` — роль не `buyer`
- `422 VALIDATION_ERROR` — не передан `X-User-Id` или невалидное тело
- `404 SKU_NOT_FOUND` — SKU не найден (`details.sku`)
- `409 INSUFFICIENT_POINTS` — недостаточно поинтов (`details.balance`, `details.required`)

### 2.2.2. Получение списка заказов
**Метод:** `GET`  
**URL:** `/orders`  
**Заголовки:** `X-Role: buyer | fulfillment_admin`  
**Доп. заголовок:** для `buyer` обязателен `X-User-Id`

**Query (опционально):**
- `status` — фильтр по статусу (точное значение, например `Новый`)
- `user_id` — фильтр по пользователю (имеет смысл для `fulfillment_admin`)

**Что делает:**
- Если `X-Role = buyer` → возвращает **только** заказы пользователя `X-User-Id`.  
- Если `X-Role = fulfillment_admin` → возвращает все заказы, можно сузить через `user_id` и/или `status`.

**Ответ:** `200 OK` и массив заказов.

**Типовые ошибки:**
- `403 FORBIDDEN` — роль не `buyer` и не `fulfillment_admin`
- `422 VALIDATION_ERROR` — роль `buyer`, но не передан `X-User-Id`

### 2.2.3. Смена статуса заказа
**Метод:** `PUT`  
**URL:** `/orders/{order_id}/status`  
**Заголовки:** `X-Role: fulfillment_admin`  
**Параметр пути:** `order_id` (int)

**Формат запроса (Body):**
```json
{ "status": "На проверке" }
```

**Какие статусы существуют:**
`Новый`, `На проверке`, `Сборка`, `Готов к выдаче`, `Завершен`, `Отменен`

**Допустимые переходы:**
- `Новый` → `На проверке` или `Отменен`
- `На проверке` → `Сборка` или `Отменен`
- `Сборка` → `Готов к выдаче`
- `Готов к выдаче` → `Завершен`

**Что происходит внутри при некоторых переходах:**
- При переходе в `На проверке` выполняется резервирование остатков по всем позициям (если не хватает — ошибка `409 INSUFFICIENT_STOCK`).  
- При переходе в `Отменен` (если разрешено) снимается резерв и возвращаются поинты.  
- При переходе в `Завершен` списываются остатки по складу.

**Ответ:** `200 OK` и обновлённый заказ (обновятся `status`, `updated_at`, добавится запись в `history`).

**Типовые ошибки:**
- `403 FORBIDDEN` — роль не `fulfillment_admin`
- `404 NOT_FOUND` — заказа нет
- `400 INVALID_STATUS_TRANSITION` — переход запрещён (`details.allowed`)
- `400 CANCEL_NOT_ALLOWED` — отмена не разрешена из текущего статуса
- `409 INSUFFICIENT_STOCK` — не хватает остатков для резерва

### 2.2.4. Отмена заказа
**Метод:** `DELETE`  
**URL:** `/orders/{order_id}`  
**Заголовки:** `X-Role: buyer | fulfillment_admin`  
**Доп. заголовок:** для `buyer` обязателен `X-User-Id`  
**Параметр пути:** `order_id` (int)

**Что делает:** переводит заказ в статус `Отменен`, но только если текущий статус — `Новый` или `На проверке`.  
При отмене сервер:
- снимает резерв (если он был),
- возвращает поинты пользователю,
- пишет событие в историю заказа.

**Ответ:** `200 OK` и заказ со статусом `Отменен`.

**Типовые ошибки:**
- `403 FORBIDDEN` — роль не `buyer`/`fulfillment_admin`
- `422 VALIDATION_ERROR` — `buyer`, но не передан `X-User-Id`
- `403 FORBIDDEN` — `buyer` пытается отменить чужой заказ
- `404 NOT_FOUND` — заказа нет
- `400 CANCEL_NOT_ALLOWED` — отмена запрещена в текущем статусе

## Сценарий тестирования и скриншоты (1–18)

Ниже описано, что делает каждый запрос, зачем он нужен и какой результат считается правильным. Под каждым шагом приложены соответствующие скриншоты.

## Почему в отчёте 18 запросов, хотя endpoint’ов 7

В Swagger описано **7 endpoint’ов** (уникальные сочетания *HTTP-метод + путь*).  
В отчёте приведены **18 запросов**, потому что здесь нумеруются не endpoint’ы, а **тестовые сценарии** (конкретные HTTP-вызовы с разными условиями: позитивный/негативный, разные роли, разные параметры).

Требование лабораторной — минимум **2 теста на каждый endpoint**, то есть при 7 endpoint’ах получается минимум **14 сценариев**.  
Количество увеличено до 18, потому что часть проверок требует **подготовить состояние системы**. Например, чтобы показать, что *отмена заказа запрещена после начала сборки*, нужно:
1) создать заказ,  
2) перевести его в «На проверке»,  
3) перевести в «Сборка»,  
4) и только после этого выполнить отмену и получить ожидаемую ошибку.

Поэтому запросы **15–17** в сценарии — это подготовка состояния (переводы по статусам), а **18** — целевой негативный тест отмены.

1) POST /products → запросы 1–2 (201 и 409)
2) GET /products → 3–4 (200 непусто и 200 пусто по поиску)
3) GET /products/{id} → 5–6 (200 и 404)
4) POST /orders → 7–8 (201 и 409)
5) GET /orders → 9–10 (200 и 403)
6) PUT /orders/{id}/status → 11–12 (200 и 403) + 16–17 (подготовка состояния “Сборка”)
7) DELETE /orders/{id} → 14 (успешная отмена) и 18 (отмена запрещена)

А 13 и 15 — это “создать заказ” специально под сценарии отмены.

Ниже описано, что делает каждый запрос, зачем он нужен и какой результат считается правильным. Под каждым шагом приложены соответствующие скриншоты.
